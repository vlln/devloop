# 6 条系统边界

不可违反的硬性约束。每条边界包含 Level 1 机制约束（不可违反）和 Level 2 实现模式（可变）。

## 边界 1：契约与执行分离

### 四类最小契约（必须存在，必须冻结后才能进入 DEVELOP）

| 契约类型 | 必须回答的问题 | 冻结时机 |
|----------|--------------|----------|
| 业务契约 | 做什么、为谁做、为什么 | DESIGN 最早冻结 |
| 接口契约 | 模块间如何通信、数据长什么样 | 业务契约冻结后 |
| 质量契约 | 什么算成功、什么算失败（AC） | 与业务契约同步冻结 |
| 架构契约 | 技术选型、约束、取舍 | 接口契约定义前冻结 |

### 冻结的硬性定义

- 冻结 = status 从 `draft` 变为 `active`，伴随独立 commit
- 冻结后不可原地修改。要改：退回 DESIGN → 新建版本 → 旧版本归档
- 复杂度等级决定契约密度，不改变冻结的强制性

### 并行化的唯一前提

接口契约 + 质量契约已冻结。此时前后端可基于同一套 Mock 和 AC 分叉并行。

## 边界 2：验证先于实现

### 分层智能化

```
低层（近代码）→ 机器化
  单元测试、接口契约校验、类型检查

高层（近意图）→ 智能化（Agent）
  E2E 语义验证、UI 合理性判断、AC 中模糊标准的判定
```

### 最小验证系统

- AC 应尽可能可量化，但不强制机器可验证
- 机器能验证的尽量机器化；机器做不到的由 Agent 在 Report 中给出判定结论
- 测试系统可以是"机器 + Agent"的组合

### 时序

编码开始时，测试基础设施（轨道 C）必须已就位。

### 验证层级（自底向上叠加，不可跳过）

```
单元测试（必选，所有项目）
  → 集成测试（必选，所有项目）
    → 接口契约测试（API 类项目）
      → 接口编排测试（多接口联动）
        → E2E 测试（有 UI 的项目）
          → 视觉回归（前端项目）
```

## 边界 3：状态变更 = 原子承诺

### 最小执行边界

- 每个 Plan 至少在一个独立 Git branch 中执行。branch 是底线。
- worktree 是并行优化手段（可选）。

### 异常处理

| 异常 | 处理 |
|------|------|
| Agent 崩溃 | 丢弃 branch/worktree，任务重置为 pending |
| 步骤失败 | branch 内回退至步骤快照，内部重试 |
| 架构不可行（DEVELOP 中发现） | 停止 DEVELOP，退回 DESIGN |
| 局部 bug（INTEGRATE 中发现） | DEVELOP ↔ INTEGRATE 修复循环 |

## 边界 4：可追溯性

### 语义链（底线）

```
Design Spec AC-003
  → Plan: "实现 AC-003 订单创建"
    → Commit: "feat(order): 实现 AC-003 订单创建"
      → Report: "AC-003 ✅, commit abc123"
```

不依赖 frontmatter 的 related 字段。

### 文档/代码 commit 分离

一次 commit 要么是文档，要么是代码。

### 准入判断（灵活）

- 程序化：检查 status 字段
- Agent 化：语义理解、上下文判断

---

## 边界 5：自描述入口

入口路径：`AGENTS.md → docs/README.md（当前状态）→ 各级 README → 具体文档`

当前状态（阶段 + 行为边界）在 `docs/README.md` 中单点可见。Agent 零上下文启动仅凭文件系统即可理解项目全貌。

---

## 边界 6：设计者意图为根

- 所有顶层目标来自 Designer 的设计意图
- Executor 不可创造新的顶层目标
- 架构变更、版本发布、需求调整由 Designer 决策
- 此边界保护职责分离，不绑定特定身份
