# DESIGN 阶段

## 固定阶段

DESIGN 是需求分析、架构设计、契约定义的阶段。**主体串行，有因果链，不可跳过。** 禁止编写业务代码。四类契约（业务/接口/质量/架构）冻结后推进到 DEVELOP。

## 子阶段

### 业务契约（必选，所有项目）

**产出：** vision.md + Design Spec（含 AC）

1. 编写 vision.md：业务目标、用户范围、顶层架构约束。一次性定稿。
2. 编写 Design Spec（参考 `docs/arch/design/001-template.md` 的结构）
3. AC 要求：尽可能可量化。机器不可验证的 AC 标注"校验方式: 手动/Agent"。

**退出条件：** vision.md status=active，Design Spec status=active。冻结后不可原地修改。

### 架构契约（必选，所有项目）

**产出：** ADR

每个技术决策一个文件（参考 `docs/arch/adr/0001-template.md` 的结构）。

**退出条件：** 所有核心 ADR status=accepted。

**不确定性处理（Spike）：**
- **技术选型类 ADR**（使用了不熟悉的库/框架/协议/依赖）：必须通过 Spike 验证，不可假设可行
- **约定/标准类 ADR**（编码规范、版本策略、命名约定）：不需要 Spike
- 验证方式：创建临时 branch，编写最小验证代码，验证后丢弃
- 验证结果记录在 ADR 的「验证」段
- 验证通过 → ADR status → accepted；验证失败 → 选择替代方案，重新验证
- 技术选型类 ADR 的「验证」段为空 → 不应标记为 accepted

### 接口契约（适用：有 API 的项目）

**产出：** API 契约定义

接口详细定义（OpenAPI 或内嵌在 Design Spec 中）。覆盖：入参、出参、错误码。

**退出条件：** API 契约已定义，所有接口有明确的入参/出参/错误码。

### 测试覆盖规划（适用：所有项目）

**产出：** AC 验证计划

AC 定稿后可与架构契约并行。设计阶段不规划单元测试（属于开发自测），只确保系统级验证可行。

**验证方式标注：** 对每条 AC 标注验证方式（自动化 / Agent 判定 / 手动）。具体测试用例由开发者在各自轨道编写。

**测试基础设施评估：** 判断计划的测试基础设施是否覆盖项目类型所需的系统级测试层（集成/E2E/视觉回归）。如果不能，调整测试基础设施 Plan。

**退出条件：** 每条 AC 有明确的验证方式，测试基础设施覆盖项目类型所需的系统级测试层。

### Plan 拆解（必选，所有项目）

**产出：** Plan 文件夹 + project-structure.md

在 `docs/plans/` 下为每个并行轨道创建文件夹。轨道数量取决于项目类型。每个文件夹内创建 README.md（子任务状态表）。Plan 文件必须包含「执行边界」段（见 phase-develop.md）。

**项目结构定义：** 拆解 Plan 前，在 `docs/arch/` 下创建 `project-structure.md`，描述：

- 顶层目录及用途（`src/`、`tests/`、`docs/`、`config/` 等）
- 模块到目录的映射（从 Design Spec 模块划分推导）
- 测试目录约定（单元测试在哪、集成测试在哪）
- 构建/配置入口位置

**不定义：** 模块内部的细粒度文件、框架脚手架生成的文件、单个源文件命名（属于 CONTRIBUTING.md 或执行者自决）。

**退出条件：** project-structure.md 已创建，所有轨道 Plan 文件夹已创建，每个包含 README.md。

### 设计变更（适用：从 DEVELOP 退回时）

**产出：** 修订后的 ADR + 更新的 Design Spec + 调整后的 Plan

1. **定级：** 判断是轻微约束还是架构颠覆
   - 轻微约束：依赖能用但方式与预期不一致 → 更新 ADR 追加修订记录，不退回 DESIGN（在 DEVELOP 内解决）
   - 架构颠覆：技术路线走不通 → 进入下方流程
2. **ADR 修订：** 旧 ADR 追加修订记录（记录原方案、实测限制、新方案），或标记 superseded + 新建 ADR
3. **传导下游：** 同步更新受影响的 Design Spec（接口、数据模型、AC）、通知受影响轨道、更新 Plan
4. **恢复编码：** 以新 ADR 为依据重新推进到 DEVELOP

**退出条件：** ADR 修订完成，Design Spec 同步更新，Plan 已调整。

## 设计完成检查

DESIGN 的出口把关是**内容级检查**，不仅检查文件是否存在。以下检查按项目类型区分「必检」和「适用」：

### 首次设计

**vision.md（必检，所有项目）：**

- [ ] status=active
- [ ] 业务目标：非空（1-2 句话即可）
- [ ] 用户范围：非空

**Design Spec（必检，所有项目）：**

- [ ] status=active
- [ ] 用户故事 ≥ 1 条（每条含编号、角色、需求、目的、优先级）
- [ ] 模块划分 ≥ 2 个模块（每个含职责说明）
- [ ] AC ≥ 1 条（每条含前置条件、操作步骤、预期结果、校验方式）
- [ ] [适用] 有 API → 接口定义（入参/出参/错误码）
- [ ] [适用] 有持久化 → 数据模型
- [ ] [适用] 前端项目 → UI 约束

**核心 ADR（必检，所有项目）：**

- [ ] 技术栈 ADR status=accepted，验证段不为空
- [ ] [适用] 有持久化 → 存储 ADR status=accepted，验证段不为空
- [ ] [适用] 多模块/多服务 → 架构模式 ADR status=accepted，验证段不为空

**接口契约（适用：有 API 的项目）：**

- [ ] 所有接口有明确的入参/出参/错误码

**测试覆盖规划（必检，所有项目）：**

- [ ] 每条 AC 有验证方式标注
- [ ] 测试基础设施评估：计划的测试基础设施覆盖项目类型所需的系统级测试层

**Plan 拆解（必检，所有项目）：**

- [ ] project-structure.md 已创建（顶层目录 + 模块映射 + 测试目录约定）
- [ ] 所有轨道 Plan 文件夹已创建，每个含 README.md
- [ ] 每个 Plan 含「执行边界」段

### 增量迭代

已有文档已冻结，仅检查**新增/修改**的文档：

- [ ] 新增 Design Spec 满足上述内容级检查
- [ ] 修改 Design Spec 的节保持原有完整性
- [ ] 新增 ADR status=accepted（技术选型类验证段不为空）
- [ ] 新增 Plan 含执行边界
- [ ] 已有 ADR 未被原地修改（如需修订，走设计变更流程）

### 设计变更

从 DEVELOP 退回时，仅检查**变更影响范围**：

- [ ] ADR 修订完成（旧 ADR 追加修订记录或标记 superseded + 新建 ADR）
- [ ] Design Spec 同步更新（受影响章节）
- [ ] Plan 已调整（受影响轨道）
- [ ] 新增技术选型类 ADR 的验证段不为空

---

全部满足后：更新 `docs/README.md` 当前阶段为 DEVELOP，追加最近事件，提交。约定前缀 `docs(state):`。

## 回退规则

- 发现架构不可行 → 重新设计，不推进
- 在其他阶段发现设计缺陷 → 更新 `docs/README.md` 当前阶段为 DESIGN，提交

## 参考实现

### 全栈 Web 项目

子阶段组合：业务契约 → 架构契约 → 接口契约 → 测试覆盖规划 → Plan 拆解

Plan 拆解为 5 条轨道：
- `0001-测试基础设施`（依赖：AC + 架构契约，必须最先完成）
- `0002-后端开发`（依赖：接口契约 + 数据模型）
- `0003-前端开发`（依赖：API Mock + UI 约束）
- `0004-部署与运维`（依赖：数据模型 + 架构契约，包含 CD 脚本、数据库 migration、环境配置）
- `0005-系统级测试用例`（依赖：AC + 测试基础设施，在 A 就位后、INTEGRATE 之前完成）

### CLI 工具项目

子阶段组合：业务契约 → 架构契约 → 测试覆盖规划 → Plan 拆解

Plan 拆解为 2 条轨道：
- `0001-测试基础设施`
- `0002-核心功能`

### 前端组件库项目

子阶段组合：业务契约 → 架构契约 → Plan 拆解

Plan 拆解为 2 条轨道：
- `0001-组件开发`
- `0002-测试基础设施`
