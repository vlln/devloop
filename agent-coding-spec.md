## 一、概述与目标

### 目标

设计一套**无人循环自主 Agent 编码系统**的文档规范，使 Agent 能够：

1. 从零接手项目时，通过文档完整了解上下文，无需人工持续 prompt
2. 在明确的权限边界内自主执行开发任务
3. 所有产出可追溯（计划→执行→报告→commit）
4. 意外中断后仅凭文件系统恢复状态，不依赖对话历史

### 适用范围

本文档定义的是**文档体系的结构和规则**。编码规范、Git 工作流等见 `skills/devpact/assets/templates/CONTRIBUTING.md`。

### 三层交付物

```
agent-coding-spec.md  ← 元设计（本文档）：描述"为什么这样设计"
skills/devpact/     ← 执行层（bootloader + 模板）：Agent 加载后知道如何操作系统
```

## 二、核心设计原则

### 2.1 渐进式披露

每层只描述**下一级**的内容，不越级展开细节。

```
skills/devpact/SKILL.md（bootloader：状态机 + 路由）
  └── AGENTS.md（项目入口地图：文档类型、目录结构、系统边界、阶段行为）
        └── docs/README.md（当前状态 + 最近事件 + 设计评估）
              ├── design/README.md（Spec 版本索引）→ 001-template.md
              ├── adr/README.md（决策索引）→ 0001-xxx.md
              └── plans/README.md（任务列表）→ 0001-xxx/README.md
```

模板格式只出现在模板文件自己所在的那一层，上层只做引用。

### 2.2 单一事实源

每类信息只有一个权威定义位置，其余文档只引用不复制。

| 信息类型 | 唯一定义位置 | 其余文档 |
|----------|-------------|----------|
| 业务需求、接口、数据模型、AC | Design Spec | 只引用，不复制 |
| 技术选型、架构折中 | ADR | 只引用，不复制 |
| 顶层愿景、硬性约束 | Vision | 只引用，不复制 |
| 编码规范、Git 规则 | CONTRIBUTING.md | 只引用，不复制 |
| 系统当前阶段 + 最近事件 | docs/README.md | 只引用，不复制 |
| 阶段行为边界 | AGENTS.md | 只引用，不复制 |

### 2.3 阶段行为模型

Agent 无固定身份，行为由当前阶段决定。状态机中的每个阶段定义了该阶段允许和禁止的行为。Agent 读 `docs/README.md` 确认当前阶段后，对照 AGENTS.md 中的阶段行为边界表行动。

**阶段行为边界（概述）：**

| 阶段 | 允许 | 禁止 |
|------|------|------|
| INIT | 创建目录结构、标准文件、初始化 Git/CI | 编写 Vision/Design/ADR/Plan、编写代码 |
| DESIGN | 编写 Vision→Design→ADR→API 契约→拆解 Plan | 编写代码、执行 Plan |
| DEVELOP | 创建 Plan、执行 Plan（编码/测试）、更新 Plan/Report | 修改 Vision/Design/ADR、无 Plan 直接写代码 |
| INTEGRATE | 执行全量测试、修复集成问题 | 新增功能代码、修改 Design/ADR |
| PRE_RELEASE | 部署到 staging、冒烟测试、环境检查、版本号确认 | 修改代码和文档 |
| RELEASE | 版本归档、CHANGELOG 整理、迭代复盘 | 修改任何代码 |

**状态变更由协调者执行。** 协调者可以是任何 Agent——它加载 skill，理解状态机，推进状态流转。协调者不固定身份，根据当前阶段切换行为模式。

### 2.4 Git 作为存储底座

Git 是整个项目（代码 + 文档）的**存储和版本控制器**。系统中所有状态流转和迭代都以 Git 动作作为切分边界：

- **一次迭代** = 隔离环境内完整执行任务 → 通过全部校验 → 原子合并入主分支。合并动作是项目进度的唯一有效变更点。
- **文档变更**必须独立 Commit，不与代码混合。
- **状态流转**伴随对应的 Git commit，commit 是状态变更的物理锚点。
- **容错**：任务异常中断 → 丢弃隔离环境，主分支零污染。误操作通过 Git 历史回退。

## 三、系统边界

以下 6 条边界是系统不可违反的硬性约束。每条边界内部有 Level 1 子边界（机制约束，不可违反）和 Level 2 实现模式（可变，按项目类型自适应）。

### 边界 1：契约与执行分离

在编写任何代码之前，契约必须冻结。这是并行化的唯一前提。

**Level 1 — 最小契约类型集：**

| 契约类型 | 必须回答的问题 | 冻结时机 |
|----------|--------------|----------|
| 业务契约 | 做什么、为谁做、为什么 | DESIGN 阶段最早冻结 |
| 接口契约 | 模块间如何通信、数据长什么样 | 业务契约冻结后 |
| 质量契约 | 什么算成功、什么算失败（AC） | 与业务契约同步冻结 |
| 架构契约 | 技术选型、约束、取舍 | 接口契约定义前冻结 |

**Level 1 — 冻结的硬性定义：**

- 冻结 = 对应文档的 `status` 字段从 `draft` 变为 `active`，伴随独立 Git commit
- 冻结后不可原地修改。要改必须走正式回退：退回 DESIGN → 新建版本 → 旧版本归档
- 复杂度等级决定契约密度（一个文件 vs 多个文件），不改变冻结的强制性

**Level 2 — 实现模式：**

- 小项目（CLI 工具）：业务契约 + 质量契约合并为一个文件
- 中项目（后端 API）：Design Spec + ADR 分离
- 大项目（全栈）：四种契约各自独立文件 + API 契约文档

### 边界 2：验证先于实现

验证标准（AC、测试用例）必须在实现之前或独立于实现存在。

**Level 1 — 验证分层智能化：**

```
低层（近代码）→ 机器化
  单元测试、接口契约校验、类型检查

高层（近意图）→ 智能化（Agent）
  E2E 语义验证、AC 中模糊标准的判定
```

**Level 1 — 最小验证系统：**

- AC 应尽可能可量化，但**不强制机器可验证**
- 机器能验证的尽量机器化；机器做不到的由 Agent 在 Report 中给出判定结论
- 测试系统可以是"机器 + Agent"的组合

**Level 1 — 验证与编码的时序：**

编码开始时，测试基础设施必须已就位。

**Level 1 — 验证层级按项目类型自底向上叠加：**

```
单元测试（必选，所有项目）
  → 集成测试（必选，所有项目）
    → 接口契约测试（API 类项目）
      → 接口编排测试（多接口联动场景）
        → E2E 测试（有 UI 的项目）
          → 视觉回归（前端项目）
```

每层都是合并闸门，不可跳过。

**Level 2 — 实现模式：**

- 机器化测试：Jest/Vitest/Playwright 等框架
- Agent 化测试：独立的测试 Agent，读取 AC 和代码产物，执行语义判断

### 边界 3：状态变更 = 原子承诺

状态从 DESIGN 变为 DEVELOP 是一个承诺——"契约已冻结，现在开始基于此契约构建"。

**Level 1 — 最小执行边界：**

- 每个 Plan 至少在一个**独立 Git branch** 中执行。branch 是底线。
- worktree 是**并行优化手段**（可选）。

**Level 1 — 变更锚点：**

| 锚点 | 含义 |
|------|------|
| commit | 状态变更的物理锚点 |
| merge | 迭代进度的原子单位 |
| 具体操作 | commit/merge/rebase/squash 由项目 Git 规范灵活决定 |

**Level 1 — 异常处理：**

| 异常 | 处理 |
|------|------|
| Agent 崩溃 | 丢弃 branch/worktree，任务重置为 pending |
| 步骤失败 | branch 内回退至步骤快照，内部重试 |
| 架构不可行 | 停止 DEVELOP，退回 DESIGN |
| 局部 bug | INTEGRATE 内部修复循环 |

**Level 2 — 实现模式：**

- 串行项目：每个 Plan 一个 branch，顺序执行
- 并行项目：多个 Plan 在独立 worktree 中并行执行

### 边界 4：可追溯性

每一次变更都必须能追溯到：谁决定的（ADR/Design）→ 谁计划的（Plan）→ 谁执行的（Report）→ 哪次 commit。

**Level 1 — 语义链（底线）：**

追溯通过文档正文中的文本引用、命名约定、commit message 关联自然形成，不依赖 frontmatter 的 `related_*` 字段：

```
Design Spec AC-003
  → Plan 01-plan-order-api.md 步骤 2: "实现 AC-003 订单创建"
    → Commit: "feat(order): 实现 AC-003 订单创建"
      → Report 01-report-order-api.md: "AC-003 ✅, commit abc123"
```

**Level 1 — 文档变更与代码变更分离：**

一次 commit 要么是文档，要么是代码，不混合。

**Level 2 — Key 化（优化）：**

frontmatter 字段、结构化元数据。按需使用。断链 = 警告，不阻塞合并。

**Level 2 — 准入判断（灵活）：**

- 程序化：简单场景，检查 `status` 字段
- Agent 化：复杂场景，语义理解、上下文判断

### 边界 5：自描述入口

任何 Agent（无论有无上下文）进入系统后，仅凭文件系统就能理解：当前处于什么阶段、能做什么不能做什么、去哪里找下一步需要的信息。

**Level 1 — 入口路径：**

```
AGENTS.md → docs/README.md（当前状态）→ 各级 README（索引）→ 具体文档
```

**Level 1 — 当前状态单点可见：**

系统当前阶段 + 最近事件 + 设计评估在 `docs/README.md` 中追踪。

**Level 1 — 零上下文启动：**

系统不依赖外部记忆或对话历史。Agent 仅凭文件系统即可理解项目全貌。

### 边界 6：设计者意图为根

系统没有自主目标。所有顶层目标的唯一来源是设计意图。

**Level 1 — 目标来源：**

所有顶层目标来自 DESIGN 阶段的设计意图。DEVELOP 阶段不可创造新的顶层目标。

**Level 1 — 决策权归属：**

架构变更、版本发布、需求调整由 DESIGN 阶段决策。

**Level 1 — 角色边界固定：**

此边界保护的是**职责分离**——设计意图不被执行过程僭越——而非特定身份。

## 四、系统生命周期状态机

系统状态机是整个开发流程的核心控制骨架。它定义了项目从初始化到发布完成的完整生命周期。

### 4.1 设计原则

- **外层串行，内层并行。** 状态机本身是串行骨架（关口），并行发生在状态内部。
- **契约是并行化的唯一前提。** 一旦接口契约 + 质量契约冻结，DEVELOP 内部各轨道可完全并行。
- **唯一串行关口：INTEGRATE。** 必须等所有并行模块联调完成才能进入。

### 4.2 三层模型

每个固定阶段内部有三层结构：

| 层次 | 含义 | 可变性 |
|------|------|--------|
| 固定阶段 | 状态机保证，所有项目必经 | 不可变 |
| 子阶段 | 按项目类型选择组合，每个标注必选/适用 | 可选组合 |
| 参考实现 | 示例，展示一种可行的覆盖方式 | 完全可变 |

### 4.3 状态总览

```
INIT ──→ DESIGN ──→ DEVELOP ──→ INTEGRATE ──→ PRE_RELEASE ──→ RELEASE
            ↑                       │  ↑
            │                       └──┘
            │                   (内部修复循环)
            │
            └──────────────────────────┘
               (架构变更回退)

RELEASE → DESIGN  (新一轮迭代)
```

6 个状态，1 个内部修复循环（INTEGRATE 内），1 个回退路径（DESIGN）。

### 4.4 各状态定义

#### INIT — 项目初始化

| 维度 | 定义 |
|------|------|
| **含义** | 项目骨架搭建。纯串行，前置环节 |
| **允许** | 创建目录结构、AGENTS.md/CONTRIBUTING.md/CHANGELOG.md、初始化 Git、初始化 CI 配置 |
| **禁止** | 编写 Vision/Design/ADR/Plan、编写业务代码 |
| **准入条件** | —（项目启动的起点） |
| **→ DESIGN 条件** | 目录结构就位、标准文件已创建、Git 仓库已初始化 |

#### DESIGN — 规划设计

| 维度 | 定义 |
|------|------|
| **含义** | 需求分析、架构设计、契约定义。主体串行（因果链），后期可有限并行 |
| **允许** | 编写 Vision → Design Spec → ADR → API 契约 → 拆解 Plan |
| **禁止** | 编写代码、执行 Plan |
| **准入条件** | INIT 完成 |
| **→ DEVELOP 条件** | 四类契约已冻结（status=active），Plan 已拆解完成 |
| **回退** | 发现架构方案不可行 → 退回 DESIGN 自身 |

**DESIGN 子阶段：**

| 子阶段 | 适用 | 产出 | 退出条件 |
|--------|------|------|----------|
| 业务契约 | 必选 | vision.md + Design Spec（含 AC） | status=active |
| 架构契约 | 必选 | ADR | status=accepted |
| 接口契约 | 适用：有 API 的项目 | API 契约定义 | 入参/出参/错误码已定义 |
| 测试覆盖规划 | 必选 | 每条 AC 标注验证方式，基础设施覆盖系统级测试层 | 全部 AC 有验证方式，基础设施覆盖到位 |
| Plan 拆解 | 必选 | Plan 文件夹 + README | 所有轨道 Plan 文件夹已创建 |

**设计评估：** 推进到 DEVELOP 前，在 `docs/README.md` 中设置评估等级：
- `完备` — 四类契约全部冻结，AC 覆盖所有已知场景
- `基本完备` — 核心 AC 已冻结，部分边缘场景标注为 Agent 判定
- `不完备` — 关键 AC 缺失或模糊，需继续完善

**DESIGN 内部因果链：**

```
Vision → Design Spec → ADR → API 契约定义 → Plan 拆解
  │                    │
  │                    └──→ 测试覆盖规划（AC 定稿后可并行）
  │
  └── 一次性定稿，几乎不修改
```

#### DEVELOP — 编码开发（最高并行期）

| 维度 | 定义 |
|------|------|
| **含义** | 基于冻结契约，多轨道并行执行。Plan 是协调者与执行者之间的契约 |
| **允许** | 创建 Plan、执行 Plan（编码/测试）、更新 Plan/Report 状态 |
| **禁止** | 修改 Vision/Design/ADR、修改全局契约 |
| **准入条件** | DESIGN 完成（四类契约已冻结，Plan 拆解完成） |
| **→ INTEGRATE 条件** | 全部并行轨道 Plan done |

**DEVELOP 子阶段：**

| 子阶段 | 适用 | 产出 |
|--------|------|------|
| 创建 Plan | 必选 | Plan 文件（含关联文档、执行边界） |
| 执行 Plan | 必选 | 代码 + 测试 |
| 检查 Report | 必选 | 验证通过 |
| 测试基础设施 | 适用：需要自动化测试 | 测试环境、框架、脚本 |

**Plan 与 Report 职责分离：**

| Plan（静态，创建后不改） | Report（动态，执行中持续更新） |
|--------------------------|------------------------------|
| 关联文档、目标、前置条件 | 执行摘要、关联 Commit |
| 依赖项、分步计划、执行边界 | 产物清单、测试摘要、验收结果 |
| — | 变更摘要、当前进度、阻塞点、遗留问题、改动建议 |

**执行边界：** Plan 必须包含「执行边界」段，明确指定执行者必须做和必须不做的事。执行者可能不加载 skill，仅凭 Plan 和系统文档工作，因此执行边界是执行者与协调者之间的契约。

**并行轨道：** 轨道数量取决于项目类型。每条轨道在独立 Git branch 中执行。测试基础设施必须最先完成。

参考实现：
- 全栈 Web：4 条轨道（后端/前端/测试基建/数据库部署）
- CLI 工具：2 条轨道（核心功能/测试基础设施）

#### INTEGRATE — 集成校验（唯一串行关口）

| 维度 | 定义 |
|------|------|
| **含义** | 整体模块联调完成，正式提测。执行系统级测试（集成/E2E/视觉回归）。单元测试已在 DEVELOP 完成 |
| **允许** | 执行全量测试、修复集成问题、生成测试报告 |
| **禁止** | 新增功能代码、修改 Design/ADR |
| **准入条件** | DEVELOP 全部轨道 Plan done |
| **→ PRE_RELEASE 条件** | 全部测试层通过 |

**INTEGRATE 内部修复循环：**

```
INTEGRATE（执行全量测试）
  ├─ 全部通过 → PRE_RELEASE
  └─ 测试失败 → 退回 DEVELOP 对应轨道修复 → 再测 → INTEGRATE
```

- 修复循环在 INTEGRATE 内部完成，不触发全局状态回退
- 仅局部 bug 在此循环内修复；若发现设计缺陷 → 退回 DESIGN
- 测试分层执行：先单模块，再跨模块集成，最后全流程 E2E

#### PRE_RELEASE — 部署验证

| 维度 | 定义 |
|------|------|
| **含义** | 部署到 staging 环境，执行冒烟测试，验证 CD 脚本和部署流程 |
| **允许** | 执行 CD 部署、冒烟测试、环境配置验证、版本号确认 |
| **禁止** | 修改代码、修改文档 |
| **准入条件** | INTEGRATE 全部测试通过 |
| **→ RELEASE 条件** | CD 部署到 staging 成功、冒烟测试通过 |

#### RELEASE — 迭代闭环

| 维度 | 定义 |
|------|------|
| **含义** | 部署到 production，打版本 tag，版本归档，迭代记录更新 |
| **允许** | 执行 CD 部署到 production、版本归档、CHANGELOG 整理、迭代复盘 |
| **禁止** | 修改任何代码 |
| **准入条件** | PRE_RELEASE 验证通过 |
| **→ DESIGN 条件** | 新一轮迭代启动 |

### 4.5 流转刚性规则

| 规则 | 说明 |
|------|------|
| 禁止跳跃 | 不允许跨状态流转（如 INIT → DEVELOP） |
| 禁止回跳 | 除 INTEGRATE 内部修复循环和 DESIGN 回退外，不允许逆向流转 |
| 架构变更强制回退 | 架构方案发生颠覆性变更 → 强制退回 DESIGN |
| 局部 bug 内部循环 | 仅局部 bug → 在 INTEGRATE 内部修复循环内解决 |
| 准入灵活 | 准入判断可以是程序（检查 status 字段）也可以是 Agent（语义理解） |

### 4.6 双层可见性

| 层面 | 使用者 | 用途 |
|------|--------|------|
| **外部调度层** | 协调 Agent（加载 skill） | 读取状态机，执行状态流转、准入检查、并行轨道分配、异常上浮 |
| **项目文档层** | 执行 Agent（不加载 skill） | 读取 `docs/README.md` 确认当前阶段，AGENTS.md 确认行为边界，Plan 文件确认执行任务 |

**当前状态追踪：** 系统当前阶段、最近事件、设计评估记录在 `docs/README.md` 中。状态变更时更新并追加最近事件。

## 五、文档体系

### 5.1 Vision（docs/vision.md）

| 维度 | 定义 |
|------|------|
| **定位** | 全局唯一顶层愿景：业务目标、用户范围、顶层架构约束、非功能指标、硬性规则 |
| **创建时机** | DESIGN 阶段 |
| **变更规则** | 仅重大业务转型才修订，每次变更独立 Git Commit |
| **生命周期** | 长期不变，几乎不修改 |

### 5.2 Design Spec（docs/design/00x-xxxx.md）

| 维度 | 定义 |
|------|------|
| **定位** | 固化落地方案 + AC 验收标准。覆盖业务契约和接口契约。所有业务需求的唯一事实源 |
| **创建时机** | DESIGN 阶段 |
| **变更规则** | 定稿后固化。重大重构时归档旧版，新建编号 |
| **生命周期** | 同时只有一个 active 版本，旧版标记 archived |
| **AC 验收标准** | 每条尽可能可量化。机器可验证的 AC 尽量机器化；机器不可验证的由 Agent 在 Report 中判定 |

### 5.3 ADR（docs/adr/000x-xxxx.md）

| 维度 | 定义 |
|------|------|
| **定位** | 架构决策记录：技术选型、方案折中、风险取舍。覆盖测试架构、部署架构、数据架构 |
| **创建时机** | DESIGN 阶段或任何需要技术决策时 |
| **变更规则** | 持续新增，不删除。状态流转：draft → accepted → superseded/deprecated |
| **生命周期** | 长期留存，永久追溯 |

### 5.4 Plan + Report（docs/plans/000x-任务名/）

| 维度 | 定义 |
|------|------|
| **定位** | 单次执行容器。Plan 是协调者与执行者之间的契约 |
| **创建时机** | DESIGN 阶段创建容器，DEVELOP 阶段填充 Plan 内容 |
| **变更规则** | Plan 创建后不改；Report 执行中持续更新，完成后标记 complete |
| **生命周期** | 短期，任务完成即完结。原地保留不删除 |

**内部结构：**

```
000x-任务名/
├── README.md              # 子任务状态表 + AC 覆盖 + commit 映射
├── 01-plan-xxx.md         # 子任务计划（静态）
├── 01-report-xxx.md       # 子任务报告（动态，一一对应）
└── artifacts/             # 执行产出
```

**Plan 段（静态）：** 关联文档、目标、前置条件、依赖项、分步计划、执行边界。

**Report 段（动态）：** 执行摘要、关联 Commit、产物清单、测试摘要、验收结果、变更摘要、当前进度、阻塞点、遗留问题、对权威文档的改动建议。

**Plan/Report 成对规则：** 一一对应。一个 Plan 可对应多个 commit。若同一 Plan 需要多次执行，续写同一个 Report。

**执行边界：** Plan 必须包含「执行边界」段，明确指定执行者必须做和必须不做的事。执行者可能不加载 skill，仅凭 Plan 和系统文档工作。

### 5.5 CHANGELOG.md（项目根目录）

| 维度 | 定义 |
|------|------|
| **定位** | 项目版本变更记录，遵循 [Keep a Changelog](https://keepachangelog.com/) |
| **创建时机** | INIT 阶段创建 |
| **变更规则** | 执行者在 Report 中记录变更摘要；RELEASE 阶段由协调者整理为正式版本条目 |
| **生命周期** | 持续更新，不删除 |

### 5.6 README.md（目录索引 + 系统状态）

| 目录 | README 内容 |
|------|------------|
| `docs/` | 子目录列表、**当前阶段**、**设计评估**、**最近事件** |
| `docs/design/` | Spec 版本列表（文件、状态） |
| `docs/adr/` | 决策列表（编号、标题、状态） |
| `docs/plans/` | 任务列表（编号、标题、状态、创建时间） |
| Plan 容器内 | 子任务状态表（编号、子任务、Plan、Report、覆盖 AC、状态、Commits） |

**docs/README.md 关键字段：**

- **当前阶段**：INIT/DESIGN/DEVELOP/INTEGRATE/PRE_RELEASE/RELEASE
- **设计评估**：完备/基本完备/不完备（仅在 DESIGN 阶段填写）
- **最近事件**：时间线日志，状态变更和 Plan 完成时追加。Agent 中断恢复时据此重建上下文

## 六、文档状态定义

每种文档类型有各自独立的状态集合和流转规则。

### 6.1 Vision

```
draft ──→ active ──→ archived
```

| 状态 | 含义 |
|------|------|
| `draft` | 草稿，尚未定稿 |
| `active` | 已定稿，当前生效 |
| `archived` | 因重大业务转型被归档 |

### 6.2 Design Spec

```
draft ──→ active ──→ archived
```

| 状态 | 含义 |
|------|------|
| `draft` | 编写中，尚未冻结 |
| `active` | 已冻结，当前唯一生效版本 |
| `archived` | 被新版本替代，原地保留 |

- 同时只有一个 active。冻结后不可原地修改。
- 归档：旧 Spec 改为 archived，更新 design/README.md

### 6.3 ADR

```
draft ──→ accepted ──→ superseded
  │                      │
  └──────────────────────┴──→ deprecated
```

| 状态 | 含义 |
|------|------|
| `draft` | 提案中，尚未采纳 |
| `accepted` | 已采纳，当前生效 |
| `superseded` | 被新 ADR 替代 |
| `deprecated` | 已废弃 |

### 6.4 Plan

```
pending ──→ in_progress ──→ done
              │
              └──→ blocked ──→ in_progress
```

| 状态 | 含义 |
|------|------|
| `pending` | 已创建，等待执行 |
| `in_progress` | 正在执行 |
| `blocked` | 遇到阻塞 |
| `done` | 已完成 |

### 6.5 Report

```
draft ──→ complete
```

| 状态 | 含义 |
|------|------|
| `draft` | 编写中，伴随 Plan 执行同步更新 |
| `complete` | 已提交 |

### 6.6 README.md / CHANGELOG.md

无 frontmatter 状态字段。

## 七、文档格式约定

### 7.1 Frontmatter

以下文档类型使用 YAML frontmatter：

| 文档类型 | 必填字段 |
|----------|----------|
| Design Spec | `title`, `description`, `type: design`, `status`, `version`, `created` |
| ADR | `title`, `description`, `type: adr`, `status`, `created` |
| Plan | `title`, `description`, `type: plan`, `status`, `created` |
| Report | `title`, `description`, `type: report`, `status`, `created` |

以下文件**不使用** frontmatter：
- AGENTS.md、CONTRIBUTING.md、CHANGELOG.md（标准文件）
- 所有 README.md（目录索引）
- vision.md（全局愿景）

字段说明：
- `title`：文档标题
- `description`：一句话摘要
- `type`：`design` | `adr` | `plan` | `report`
- `status`：见 [六、文档状态定义](#六文档状态定义)
- `created`：ISO 8601 格式
- `version`：仅 Design Spec，整数递增

### 7.2 路径引用

所有跨文档引用使用相对路径 Markdown 链接。禁止使用反引号包裹的纯文本路径。

### 7.3 命名规范

| 文档 | 命名 | 示例 |
|------|------|------|
| Vision | `vision.md` | `vision.md` |
| Design Spec | `00x-xxxx.md` | `001-vagent.md` |
| ADR | `000x-xxxx.md` | `0001-db-choice.md` |
| Plan 文件夹 | `000x-简短描述` | `0001-订单模块` |
| Plan 子任务 | `0x-plan-xxx.md` | `01-plan-order-api.md` |
| Report | `0x-report-xxx.md` | `01-report-order-api.md` |

## 九、联动规则

当修改权威文档时，需遵循以下规则：

### 修改 Design Spec

1. 更新 `docs/design/README.md` 中的版本状态
2. 检查 `docs/adr/README.md` 中所有引用该 Spec 的 ADR，更新状态
3. 若 AC 变更，生成对应 Plan 更新测试
4. 独立 Commit

### 新增/修改 ADR

1. 更新 `docs/adr/README.md`
2. 若替代旧 ADR，旧 ADR 状态改为 `superseded`，新 ADR 正文中引用旧 ADR 路径

### 归档旧 Design

1. 旧 Spec 状态标记为 `archived`（原地保留，不移动）
2. 更新 `docs/design/README.md`
3. 更新 `docs/adr/README.md` 中所有引用

## 十、测试体系（可伸缩）

### 10.1 核心原则

- 测试从**单元测试**（最内层）开始，向外逐层按需叠加
- 每层的测试用例必须绑定 Design Spec 的 AC
- 测试输出做摘要化处理，避免占用 Agent 上下文
- 测试作为合并闸门：全部测试通过方可提交合并
- 低层机器化，高层智能化

### 10.2 按项目类型伸缩

| 项目类型 | 测试层（从内到外） |
|----------|-------------------|
| 纯 CLI / 脚本工具 | 单元 → 集成 |
| 后端 API 服务 | 单元 → 集成 → 接口契约 → 接口编排 |
| 全栈 Web 应用 | 单元 → 集成 → 接口契约 → 接口编排 → E2E → 视觉回归 |
| 前端组件库 | 单元 → 组件测试 → 视觉回归 |

### 10.3 各层定位

| 层级 | 测试对象 | 执行方式 |
|------|----------|----------|
| 单元测试 | 纯函数、工具、状态计算、hooks | 机器化 |
| 集成测试 | 多单元组合的内部流程链路 | 机器化 |
| 组件测试 | 组件渲染、属性变化、事件触发 | 机器化 |
| 接口契约测试 | 单个接口的格式、参数、返回结构 | 机器化 |
| 接口编排测试 | 多接口调用时序与状态联动 | 机器化 |
| E2E 测试 | 完整用户交互流程 | 机器化 + Agent 化 |
| 视觉回归测试 | 像素差异比对 | 机器化 |

### 10.4 在文档体系中的分布

| 层级 | 内容 | 位置 |
|------|------|------|
| 验收标准 | 每条功能的可量化判定条件 | Design Spec AC |
| 测试架构 | 框架选型、分层策略、覆盖率标准 | ADR |
| PR 检查 | 提交前必须通过测试 | CONTRIBUTING.md |
| 前端约束 | 组件规范、视觉回归 | Design Spec（前端项目） |

## 十一、架构分层

### 11.1 Skills 层（bootloader）

`skills/devpact/` 是执行层入口。Agent 加载 skill 后获得：
- 状态机全貌和路由
- 系统安装指令
- 按阶段的操作流程（references/）

Skill 不包含项目规则（文档类型、命名规范、frontmatter 等）——这些在系统文档中。

### 11.2 项目文档层（模板交付物）

安装到项目中的文档。包含 AGENTS.md、CONTRIBUTING.md、CHANGELOG.md 和 docs/ 子目录。模板位于 `skills/devpact/assets/templates/`。项目内 Agent 读取此层即可工作，无需加载 skill。

### 11.3 调度/协调层（外部基础设施）

由更高层级的调度 Agent 执行。职责：读取状态机，执行状态流转、准入检查、任务分配、隔离环境创建与回收、异常上浮。项目内 Agent 不可见此层。

### 11.4 三层交互

```
调度/协调层（外部）
  │ 加载 skill → 获取状态机定义
  │ 读取 docs/README.md → 获取当前状态
  │ 执行准入检查 → 推进状态 → 更新 docs/README.md
  │ 创建隔离环境，分配执行 Agent
  ▼
Skills 层（bootloader）
  │ 提供系统安装、导航、路由
  │ 按阶段提供操作流程
  ▼
项目文档层
  │ Agent 读取 AGENTS.md → 阶段行为边界
  │ Agent 读取 docs/README.md → 当前阶段 + 最近事件
  │ Agent 读取 Plan → 执行边界 → 执行
  │ 产出通过 Git commit 反馈
```

## 十二、关键设计决策

| 决策 | 理由 |
|------|------|
| 6 条系统边界为不可违反的硬性约束 | Level 1 约束机制，Level 2 允许自适应 |
| 契约与执行分离 | 并行化的唯一前提。冻结后不可原地修改 |
| 验证先于实现 | 低层机器化，高层智能化。测试武器在编码前就位 |
| 状态变更 = 原子承诺 | branch 为最小执行边界，worktree 为并行优化 |
| 语义链为底线，Key 化为优化 | 准入判断灵活：程序或 Agent 均可 |
| 设计者意图为根 | 保护职责分离，非特定身份 |
| 系统状态机为核心控制骨架 | 6 状态 + 内部并行轨道 + 修复循环 |
| 固定阶段 → 子阶段 → 参考实现三层模型 | 固定约束 + 可选组合 + 自由实现 |
| 阶段行为模型取代角色模型 | Agent 无固定身份，行为由当前阶段决定 |
| Plan 是协调者与执行者之间的契约 | 执行者不加载 skill，仅凭 Plan 和系统文档工作 |
| Plan 静态 / Report 动态 | 计划创建后不改，执行信息在 Report 中持续更新 |
| 执行边界 | Plan 必须明确指定执行者必须做和必须不做 |
| 设计评估（完备/基本完备/不完备） | 非二元判断，允许"基本完备"进入 DEVELOP |
| 执行日志（最近事件） | 状态变更和 Plan 完成时追加，Agent 中断恢复时重建上下文 |
| AGENTS.md = 项目入口地图 | 文档类型、目录结构、系统边界、阶段行为 |
| CONTRIBUTING.md = 编码/Commit/文档/测试规范 | 统一入口，符合社区惯例 |
| Plan/Report 成对 | 每次修改有计划→执行→报告闭环 |
| Report 挂钩 commit | 可追溯 |
| 测试分层不集中 | AC 在 Design，架构在 ADR，PR 流程在 CONTRIBUTING |
| 测试按项目类型可伸缩 | 不强制固定层数 |
| 每种文档有独立的状态集合和流转规则 | 不同生命周期语义，不共用 |
| 无全局 Index.md | 各目录 README.md 自治，渐进式披露 |
| Plan 不移动归档 | README 状态字段区分，原地保留 |
| 统一 YAML frontmatter | 元信息结构化 |
| 标准文件/README 无 frontmatter | 标准文件和索引文件无需 frontmatter |
| 不设计 tag/related 字段 | 关联通过文本引用表达 |
| 相对路径链接 | 可点击、可追溯 |
| 三层交付物分离 | agent-coding-spec（元设计）/ skills（执行层）/ template（交付物） |
| 调度层与文档层分离 | 调度逻辑属外部基础设施 |
| 系统状态 + 最近事件在 docs/README.md | 单点追踪，中断恢复 |
| Git 作为存储底座 | 状态流转以 commit/merge 为切分边界 |
